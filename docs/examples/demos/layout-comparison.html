Layout Approaches Comparison

Compare three layout implementation approaches in Vanilla Breeze: Custom Elements, Data Attributes, and Grid Identity.

Three Approaches

1. Custom Elements

<layout-stack>, <layout-sidebar>, etc.

Pros: Self-documenting, encapsulated

Cons: Non-semantic wrappers, more DOM nodes

2. Data Attributes

data-layout="stack", data-layout="sidebar"

Pros: Works on any element, semantic HTML

Cons: Verbose for complex layouts

3. Grid Identity

data-layout="body-holy-grail" with semantic children

Pros: Pure semantic HTML, CSS handles mapping

Cons: Fixed element roles, body-level only

Feature Comparison

Feature Custom Elements Data Attributes Grid Identity
Semantic HTML ✗ Wrapper elements ✓ Any element ✓ Pure semantic
Self-documenting ✓ Tag name is layout ✓ Attribute is layout ✓ Structure is layout
Component-level use ✓ Anywhere ✓ Anywhere ✗ Body-level only
:has() adaptation ✗ Manual changes ✓ Auto-adapts ✓ Auto-adapts
Template tokens ✗ Not applicable ✗ Not applicable ✓ --tpl-* variables
Grid areas ✗ Flexbox-based ✓ Named areas ✓ Auto-assigned
Nesting depth ✗ Deep (wrappers) ✓ Shallow ✓ Flat

When to Use Each

Custom Elements

Best for:

  • Small, isolated layouts
  • When you want visual nesting
  • Teams familiar with Web Components

Data Attributes

Best for:

  • Component-level layouts (cards, panels)
  • Layouts on semantic elements
  • Most flexible approach

Grid Identity

Best for:

  • Full page layouts
  • App shells, dashboards
  • Maximum semantic purity
