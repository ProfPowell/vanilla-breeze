# i18n: Current Status and Consolidated Plan

Status date: February 24, 2026

This document consolidates:
- `r-n-d/i18n/i18n-overview.md`
- `r-n-d/i18n/review-1.md`
- `r-n-d/i18n/review-2.md`

and aligns them with what Vanilla Breeze actually implements today.

## 1) What Is Implemented Today

## CSS and HTML-level i18n (good baseline)

1. Script-aware typography is implemented in core CSS:
- `src/native-elements/i18n/styles.css`
- Imported by `src/native-elements/index.css`, so it ships in core CSS bundles.

2. `:lang()`-based rules exist for:
- CJK (`zh`, `ja`, `ko`)
- Arabic script (`ar`, `fa`, `ur`, `ps`)
- Hebrew (`he`, `yi`)
- Thai (`th`)
- Devanagari (`hi`, `mr`, `sa`, `ne`)

3. Ruby visibility system exists:
- `data-ruby="show|hide|auto"` on `<html>`
- `auto` re-shows ruby for CJK language contexts.

4. Locale quote styling for `<q>` is implemented via CSS `quotes`.

5. Direction helpers exist:
- `[dir="rtl"] { unicode-bidi: embed; }`
- `[dir="auto"] { unicode-bidi: isolate; }`

## JS i18n utilities (implemented)

1. Core utilities exist in `src/lib/i18n.js`:
- `getLocale()`
- `primarySubtag()`
- `isRTL()`
- `usesRuby()`
- `VbI18n` class (`t`, `apply`, `load`, `setLocale`, `watch`)

2. Locale resolution currently is:
1. `<html lang>`
2. `navigator.language`
3. `'en'`

3. `VbI18n` is app-owned:
- exported utility
- not auto-initialized
- works with `data-i18n`, `data-i18n-attr`, `data-i18n-vars`

4. Locale-change event in runtime is:
- `vb:locale-change`

## Locale-aware formatting (implemented)

1. Format utilities all use `getLocale()`:
- `src/utils/format-number-init.js`
- `src/utils/format-date-init.js`
- `src/utils/format-bytes-init.js`
- `src/utils/ticker-init.js`

2. They also support per-element overrides via `data-locale`.

## Documentation (implemented, with some drift)

1. Primary i18n doc page exists:
- `11ty-site/src/pages/docs/i18n.njk`

2. It accurately documents:
- script typography
- ruby visibility
- locale fallback chain
- `VbI18n` usage

## 2) What From the Original i18n Notes Is Outdated

## `<vb-settings>` ownership model is not current reality

The overview assumes a `<vb-settings>` runtime controller and `vb:settings-change`.
Current codebase has neither:
- No `<vb-settings>` component in `src/`
- No `vb:settings-change` event in runtime
- Existing settings UI is `settings-panel` and `theme-picker`, focused on theme/a11y/extensions, not locale/dir/ruby.

So today i18n state is rooted in:
- `<html lang>`
- optional per-element `data-locale`
- optional app-level `VbI18n` instance

not a global VB locale controller.

## Ruby annotation runtime is not shipped in core

`r-n-d/i18n/ruby-annotate.js` is proposal code, not integrated in `src/` runtime bundles.
Current shipped behavior supports ruby display/styling of existing markup, but not automatic annotation generation.

## 3) Confirmed Gaps and Risks

## P0 correctness/documentation gaps

1. Translation-protection bug:
- `src/native-elements/i18n/styles.css` uses `translate: no;`
- This is not valid for translation control (it is a transform property context, not HTML translation metadata).
- Browsers/translation tools require HTML `translate="no"` in markup.

2. Docs drift on this point:
- `docs/i18n.njk` says VB "automatically sets translate: no" for code-ish elements.
- That does not provide reliable translation protection.

3. Architecture drift:
- i18n overview centers around `<vb-settings>` and `vb:settings-change`, but current runtime does not.

## P1 robustness and performance gaps

1. `VbI18n.apply()` has unguarded JSON parsing for `data-i18n-vars`.
- Malformed JSON can throw and break the pass.

2. `VbI18n.apply()` text-node replacement is fragile for mixed-content elements.
- Only first text node is replaced.

3. Locale map precedence in `t()` is region-fragile.
- `primarySubtag(locale)` lookup happens before full-locale lookup.
- This can pick `pt` when `pt-BR` should win.

4. Intl formatter caching is not implemented.
- `Intl.NumberFormat`, `Intl.DateTimeFormat`, `Intl.RelativeTimeFormat` objects are created ad hoc.
- Not always a bottleneck, but avoidable overhead for high-volume formatting scenarios.

5. Format utilities do not auto-refresh on locale-change events.
- Existing formatted nodes do not re-render when locale changes unless reprocessed manually.

## P2 tooling/testing gaps

1. No dedicated automated i18n test coverage found for:
- `VbI18n` swap behavior
- locale fallback correctness
- script/ruby/quotes CSS behavior
- dynamic locale change behavior

2. No lint/policy layer enforcing logical-property claims for i18n-related styles.

## 4) Consolidated Direction (Keep + Fix)

## Keep

1. Platform-first model:
- `lang`, `dir`, `translate`, `Intl.*`, ruby elements, logical properties.

2. Three-layer ownership from the overview:
- Structure: framework CSS/HTML behavior
- Behavior: locale-aware formatting utilities
- Content: app-owned translation strings and strategy

3. Lightweight `VbI18n` approach instead of forcing a heavy translation framework.

## Fix

1. Correct translation-protection implementation and docs:
- remove/replace `translate: no` claim and rely on HTML `translate="no"` conventions.

2. Align architecture docs with runtime reality:
- stop referencing `<vb-settings>` as current implementation
- document current locale sources and event behavior explicitly.

3. Harden `VbI18n`:
- safe parse for `data-i18n-vars`
- deterministic locale precedence (exact locale first, then primary, then fallback)
- clearer contract for mixed-content nodes.

4. Add formatter caching and optional locale-change re-render hooks for formatting utilities.

## 5) Priority Plan

1. P0: Correct docs/runtime contract
- Fix `translate` guidance and implementation narrative.
- Replace `<vb-settings>` references with current runtime model in i18n docs.

2. P0: Ship small `VbI18n` robustness fixes
- guarded JSON parsing
- exact-locale precedence before primary-subtag fallback.

3. P1: Add formatter cache utility for Intl objects.

4. P1: Add optional global locale refresh API/event hookup for formatters.

5. P1: Add tests for `VbI18n` and locale formatting behavior.

6. P2: Re-evaluate ruby auto-annotation proposal as optional add-on, not core default.

## 6) Recommendation

The i18n foundation is already strong at the CSS/platform layer. The main work now is contract alignment and runtime hardening, not a redesign:

1. fix translation metadata correctness,
2. align docs with actual architecture,
3. make `VbI18n` safer,
4. add lightweight performance and testing improvements.
