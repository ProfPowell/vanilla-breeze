<ai-chat>

A web component wrapping Chrome's on-device LanguageModel API (Gemini Nano, stable in Chrome 148). Light DOM, data-*-driven state, configured via attributes.

Usage

<ai-chat
  system="You are a helpful, concise assistant."
  placeholder="Ask me anything…">
</ai-chat>

Or for longer system prompts, use a child <template data-role="system">. The component reads either; the attribute wins if both are present.

API surface used

  • LanguageModel.availability() on connect — drives the status dot & data-state.
  • LanguageModel.create({ initialPrompts, monitor }) — created lazily on first send (user activation requirement). The monitor reports model download progress.
  • session.promptStreaming(text, { signal }) — streams deltas into the assistant message; AbortController wires the Stop button.
  • session.contextUsage / session.contextWindow — refreshed after each turn, shown in the header.
  • contextoverflow event — emitted as a custom ai-context-overflow event so host code can react.
  • session.destroy() — called on reset and component disconnect to free the model.

Custom events

The component is a closed black box for state, but emits everything you'd want to listen for:

  • ai-state — { state }, fires on every data-state change.
  • ai-message — { role, text }, fires once per completed message (user or assistant).
  • ai-context-overflow — bubbled from the underlying session event.
  • ai-error — { error }, fires when a prompt fails.

Design notes

  • Light DOM. No shadow root — full styleability from the host page, no slotting gymnastics. Matches the <mark-down> pattern.
  • State as attributes. All chat state surfaces as data-state on the host, plus data-role on each message. CSS does the rest — show/hide Send vs Stop, dim disabled inputs, the streaming caret. No .classList juggling.
  • Lazy session. availability() runs on connect (no user gesture needed), but create() is deferred until the first send so the user activation requirement is satisfied naturally.
  • Streaming as deltas. Each chunk from promptStreaming() is concatenated into the active message. The blinking caret pseudo-element is gated on [data-streaming] on the message, so it disappears the moment the stream closes.
  • Tiny markdown. Four regex passes — escape, fenced code, inline code, bold. No library. Anything more belongs in <mark-down>, which this could nest inside if you want.
  • Field-sizing on the textarea. CSS field-sizing: content grows the input as you type — no JS resize logic.

What's deferred

  • Multimodal input (image/audio). The API supports it via expectedInputs; this POC is text-only.
  • responseConstraint for structured output. Easy add as an attribute or per-call option.
  • Persistence. initialPrompts can rehydrate a session, but you'd need to store the turns yourself.
  • session.clone() for branching conversations. The hook is there; the UI isn't.
