Classes, Data Attributes, ARIA, and Native HTML

A decision guide for expressing state, behavior, and visual variation on HTML elements — with concrete examples across a range of UI patterns.

The Core Principle

Every piece of information you attach to an HTML element has a natural home. Using the wrong container causes three problems: you lose semantics the browser would have given you for free, you create accessibility work for yourself, and your CSS becomes brittle class-name logic instead of meaning-driven selectors.

The hierarchy, from most to least preferred:

  1. 1Native HTML attributedisabled, open, selected, required, checked — the platform already named this concept
  2. 2ARIA attributearia-current, aria-expanded, aria-pressed, aria-sort — standardised semantic state without a native attribute
  3. 3data-* attributeYour custom state or configuration that has no platform equivalent
  4. 4ClassVisual variants only — things with no meaning beyond appearance

A useful test: if you removed the styling entirely, would this attribute still carry meaning? If yes, it belongs in native HTML or ARIA. If the answer is purely visual, a class is fine.

Example 1 — Navigation current page

The classic case. "Which link represents where I am right now?" is a semantic question the platform already answered.

ARIA Use aria-current="page"

❌ Wrong — class encodes meaning

<nav>
  <a href="/">Home</a>
  <a href="/about" class="active">About</a>
  <a href="/work">Work</a>
</nav>

/* CSS has to know about a made-up word */
a.active { … }

✅ Right — ARIA expresses meaning

<nav>
  <a href="/">Home</a>
  <a href="/about" aria-current="page">About</a>
  <a href="/work">Work</a>
</nav>

/* CSS reads intent, not jargon */
[aria-current="page"] { … }

Bonus: Screen readers announce "About, current page" automatically. You get that for free with zero extra work.

Example 2 — Accordion / disclosure

Open or closed is a binary state. The platform has this covered twice: natively with <details>, and semantically with aria-expanded for custom patterns.

Native Prefer <details open> when possible

❌ Wrong — JS + class for binary state

<button class="accordion-toggle open"
        onclick="toggle(this)">
  What is a widget?
</button>
<div class="accordion-panel open">
  <p>A small self-contained UI component.</p>
</div>

✅ Right — native element, zero JS

<details>
  <summary>What is a widget?</summary>
  <p>A small self-contained UI component.</p>
</details>

/* Style with native pseudo or attribute */
details[open] summary { color: var(--primary); }
details:not([open]) p { display: none; }

Live demo — no JavaScript

What is progressive enhancement?

Building in layers: HTML first, CSS for presentation, JavaScript only for what neither can handle alone.

When you can't use <details> (e.g. animated panels, tab sets): use aria-expanded="true|false" on the trigger, toggled by a small JS attribute flip — never a class.

ARIA aria-expanded for custom accordion

❌ Wrong

<button class="open" onclick="…">Section</button>
<ul class="panel visible">…</ul>

.panel { display: none; }
.panel.visible { display: block; }

✅ Right

<button aria-expanded="false"
        onclick="this.setAttribute(
          'aria-expanded',
          this.getAttribute('aria-expanded')
            === 'true' ? 'false' : 'true')">
  Section
</button>
<ul>…</ul>

[aria-expanded="true"] + ul { display: block; }
[aria-expanded="false"] + ul { display: none; }

Example 3 — Toggle button (pressed state)

A button that switches between on and off is not about appearance — it has a meaningful state: pressed or not pressed. ARIA names that precisely.

ARIA aria-pressed on toggle buttons

❌ Wrong

<button class="btn-toggle active">
  Bold
</button>

.btn-toggle.active { background: var(--primary); }

✅ Right

<button aria-pressed="true"
        onclick="this.setAttribute(
          'aria-pressed',
          this.getAttribute('aria-pressed')
            === 'true' ? 'false' : 'true')">
  Bold
</button>

[aria-pressed="true"] { background: var(--primary); }

Live demo

Screen readers announce "Bold, toggle button, not pressed" and "Italic, toggle button, pressed". The visual and semantic states are one.

Example 4 — Form field states

Forms are perhaps the richest native semantic layer. required, disabled, readonly, and browser-driven :valid/:invalid pseudo-classes replace entire categories of class juggling.

Native Native attributes + CSS pseudo-classes

❌ Wrong — classes mirror browser state

<input type="email"
       class="input input--error"
       aria-describedby="err"/>
<input type="text"
       class="input input--disabled"
       tabindex="-1" readonly/>

.input--error { border-color: red; }
.input--disabled { opacity: 0.5; }

✅ Right — the browser tracks state

<input type="email" required/>
<input type="text" disabled/>

/* CSS reads built-in pseudo-classes */
input:invalid { border-color: var(--bad); }
input:disabled { opacity: 0.5; }
input:required { outline: … }

Live demo — type in each field

Border turns green when valid, red when invalid — zero JS.

Example 5 — Dialog open / closed

The <dialog> element manages its own open state via the open attribute, set by .showModal() and .close(). You never need a class.

Native <dialog open> + commandfor

❌ Wrong

<div class="modal modal--visible">
  <div class="modal__overlay"></div>
  <div class="modal__box">
    <button onclick="closeModal()">✕</button>
    <p>Content</p>
  </div>
</div>

.modal { display: none; }
.modal--visible { display: flex; }

✅ Right

<button commandfor="my-dialog"
        command="show-modal">
  Open
</button>

<dialog id="my-dialog">
  <p>Content</p>
  <button commandfor="my-dialog"
          command="close">
    Close
  </button>
</dialog>

/* Style by element + native state */
dialog[open] { … }
dialog::backdrop { … }

Live demo

This is a native <dialog>. Focus is trapped, backdrop is automatic, Escape closes it.

Fallback note: commandfor/command is still gaining browser support. Until then, the .showModal()/.close() JS API is the correct approach — still no classes needed.

Example 6 — Sortable table columns

aria-sort is purpose-built for this. It communicates sort direction to assistive technology and gives CSS a clean hook.

ARIA aria-sort="ascending|descending|none"

❌ Wrong

<th class="sorted-asc" onclick="sort('name')">
  Name ↑
</th>

th.sorted-asc::after  { content: " ↑"; }
th.sorted-desc::after { content: " ↓"; }

✅ Right

<th aria-sort="ascending"
    onclick="cycleSort(this)">
  Name
</th>

th[aria-sort="ascending"]::after  { content: " ↑"; }
th[aria-sort="descending"]::after { content: " ↓"; }

Example 7 — Where data-* genuinely belongs

Now the case where data-* is correct: state or configuration that has no platform name. Theme preference, async loading state, and component variant type are all custom concepts.

data-* Custom state with no platform equivalent

✅ These are correct uses of data-*

<!-- Theme: not in HTML spec, your concept -->
<html data-theme="dark">
[data-theme="dark"]  { background: #111; color: #eee; }

<!-- Async loading: not a native state -->
<section data-loading>…</section>
[data-loading] { opacity: 0.5; pointer-events: none; }

<!-- Card variant: purely your design system -->
<article data-variant="warning">…</article>
[data-variant="warning"] { border-color: var(--warn); }

The tell: If you're inventing a term (data-loading, data-theme, data-variant), data-* is right. If the platform already has a word for it (open, disabled, aria-current), use that instead.

Example 8 — Where class genuinely belongs

Classes are for purely visual variation with no semantic meaning. A .lead paragraph, a .sr-only visually-hidden element, a .prose typography scope — these are presentational, not state.

Class Visual-only variation

✅ Correct uses of class

<!-- Typography scope: visual only -->
<article class="prose">…</article>

<!-- Screen-reader utility: purely presentational -->
<span class="sr-only">(opens in new tab)</span>

<!-- Layout helper: no meaning without CSS -->
<section class="grid-2col">…</section>

<!-- NOT this: "active" carries meaning → use aria-current -->
<!-- NOT this: "open" carries meaning → use [open] or aria-expanded -->
<!-- NOT this: "disabled" carries meaning → use [disabled] -->

Decision table

Situation Use Because
Current nav link aria-current="page" ARIA names "where you are" precisely
Disclosure open/closed <details open> or aria-expanded Native element or standardised state name
Toggle button on/off aria-pressed="true|false" ARIA names pressed state precisely
Tab panel selected aria-selected="true|false" ARIA names selection precisely
Table sort direction aria-sort="ascending|descending" ARIA names sort state precisely
Dialog visible <dialog> [open] Native element manages its own state
Field disabled disabled attribute Native attribute, browser enforces semantics
Field invalid :invalid pseudo-class Browser tracks validity natively
Dark/light theme data-theme="dark" No platform name for this concept
Async loading data-loading (boolean) No platform name for this concept
Design variant (warning, error) data-variant="warning" No platform name; it's your design system
Typography scope .prose class Purely visual, no semantic meaning
Visually hidden .sr-only class Presentation technique, not semantic state
Current link → "active" class ❌ .active class Invented name; aria-current already exists
Open → "open" class ❌ .open class Duplicates native [open] or aria-expanded
Selected → "selected" class ❌ .selected class Duplicates aria-selected or :checked

The test to run in your head

  1. Does HTML have a native attribute or element for this? → Use it.
  2. Does ARIA have a named attribute for this state? → Use it.
  3. Is this genuinely your own invented concept with no platform name? → data-*.
  4. Is it purely visual with no meaning if CSS is removed? → Class.

Every .active, .open, .selected, or .disabled class in a codebase is a missed step 1 or 2. The platform already had the vocabulary — it just wasn't consulted.
