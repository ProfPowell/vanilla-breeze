The Future of CSS Architecture

How native CSS features are replacing the tooling we thought we needed.

Thomas Powell · Apr 13, 2026 · 8 min read

For years, CSS architecture meant choosing between methodologies: BEM, SMACSS, OOCSS, ITCSS, Atomic CSS. Each offered structure where the language itself provided none. We adopted preprocessors like Sass for variables, nesting, and mixins. We built design systems on utility-first frameworks. We accepted the complexity as necessary.

But CSS has changed. Native features now cover most of what we once needed tools for. Custom Properties replaced variables. Native nesting arrived. @layer gave us explicit cascade control. Container queries freed us from viewport-only responsive design. :has() eliminated the need for parent selectors. :where() and :is() simplified specificity management.

The Readability Triad

Good typography rests on three legs: measure (line length), text size, and line spacing. Change one and you must adjust the others. The web gives us tools for all three — max-inline-size for measure, rem for size, and unitless line-height for spacing.

The key insight is using em for local sizing within components and rem for global page-level sizing. This prevents the classic nested-list shrinking problem while keeping components self-contained.

Good web typography does not require ornamental excess. It requires making dozens of small decisions that together let the text carry less friction.

Layer Architecture

CSS @layer provides the cascade control that methodologies tried to enforce through naming conventions. By declaring layers explicitly — tokens, reset, elements, components, utilities — you get deterministic specificity without fighting the cascade.

This is especially powerful for theme systems. A theme layer can override any base style without specificity hacks, and utility classes can always win regardless of source order.

Progressive Enhancement in Practice

The modern approach is platform-first: use native HTML elements and CSS features as the foundation, enhance with JavaScript only where the platform falls short. This isn't a limitation — it's a strength. Native elements give you accessibility, keyboard navigation, and form participation for free.

Custom elements extend HTML's vocabulary without replacing it. A <brand-mark> element is still an inline element that works without JavaScript. A <selection-menu> enhances text selection without preventing it.
